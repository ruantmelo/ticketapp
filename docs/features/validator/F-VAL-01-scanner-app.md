# F-VAL-01 — Scanner App

| Field | Value |

| **Persona** | Validator / Door Staff |

## Summary

A mobile application (React Native / Expo) used by door staff to scan the dynamic QR codes generated by buyer tickets. The app verifies the QR signature, checks on-chain ownership and burn status, and displays an instant pass/fail result. It abstracts all blockchain complexity — the staff member only sees "Valid / Already Used / Invalid" and a visual + haptic cue.

## User Story

As a door staff member, I want to scan a attendee's phone and instantly see whether their ticket is valid, so that I can admit people quickly without any blockchain knowledge and without fear of letting in someone with a fake or already-used ticket.

## Acceptance Criteria

- [ ] The app opens directly to a scanner view (one tap to start scanning).
- [ ] The camera reads the dynamic QR code (F-BUY-04) in real-time.
- [ ] On scan, the app performs the following checks in order:
  - [ ] **Signature verification**: recover the signer address from the QR signature; compare to the on-chain `ownerOf(tokenId)`. Mismatch = Invalid.
  - [ ] **Time window**: verify the QR's timestamp is within the current valid window (tolerance per ADR 0009). Expired = Invalid (stale screenshot).
  - [ ] **Burn status**: query the contract to check if the ticket has been burned. Burned = Already Used.
  - [ ] **Ownership**: confirm the signer is the current on-chain owner. Not owner = Invalid.
- [ ] The result is displayed in under 1 second of scanning.
- [ ] Results are color-coded: green (Valid), red (Invalid), amber (Already Used).
- [ ] Haptic feedback accompanies each scan result (vibration pattern per result type).
- [ ] On a Valid scan, the app triggers the burn transaction on-chain to invalidate the ticket (or queues it — see Open Questions).
- [ ] The app displays a running count of scanned attendees (valid count) for the current session.
- [ ] The app requires no blockchain configuration from the staff member — the event/contract is pre-loaded or selected from a simple list.
- [ ] The scanner works in varying lighting conditions (camera torch toggle available).

## On-Chain / Off-Chain Split

| Layer | Responsibility |
|-------|----------------|
| **On-chain** | Provides `ownerOf(tokenId)` and burn status via view functions. Accepts the burn transaction to invalidate a used ticket. |
| **Off-chain (mobile app)** | Camera/QR scanning, signature recovery (locally, using viem or ethers), on-chain state queries (read calls), burn transaction submission, result display, haptic feedback, session counting. |
| **Off-chain (backend)** | Provides the event/contract configuration to the app (which contract addresses correspond to which event). May relay burn transactions if the app does not have a direct signer. |

## Dependencies

- **F-BUY-04** (Dynamic Ticket QR) — the scanner validates the QR format and signature scheme produced by this feature.
- **F-ORG-02** (Minting Engine) — the contract being queried is deployed by the minting engine.
- **ADR 0009** (Dynamic QR signing scheme) — **blocking dependency**. The scanner must implement the exact same signature verification and time-window logic as the buyer app. Both sides must agree on the scheme.
- **F-VAL-02** (Offline Mode / Cache) — extends this app to work without internet; tightly coupled.

## Open Questions

- Who signs the burn transaction? Options: (a) the scanner app has a dedicated "validator wallet" authorized by the contract to burn tickets, (b) the backend relays burn transactions using a platform-controlled wallet, (c) the burn is batched and submitted post-event. Option (a) is fastest but requires key management on the device; (b) requires connectivity; (c) allows re-entry fraud within the event. This is a critical design decision.
- Should the app support scanning multiple events in one session (e.g., a multi-day festival), or is it locked to one event per session?
- How is the validator authenticated in the app? Options: a one-time event code shared by the organizer, QR-based login, or username/password. Should be simple for non-technical staff.
- What happens if a scan is valid but the burn transaction fails (network issue)? Should the app still admit (and queue the burn for later) or deny? This intersects with F-VAL-02.
